# Пользователи и аккаунты работников

## Общая архитектура

Система использует разделение данных между **Keycloak** (аутентификация) и **Prisma/PostgreSQL** (бизнес-данные).

### Что хранится в Keycloak

- **User ID** (`keycloakId`) - уникальный идентификатор пользователя в Keycloak
- **Email** - адрес электронной почты (используется для входа)
- **Password** - хешированный пароль
- **Email verification status** - статус подтверждения email

### Что хранится в Prisma (PostgreSQL)

- **keycloakId** - связь с Keycloak (уникальный идентификатор)
- **Email** - синхронизируется с Keycloak
- **Бизнес-данные** - все остальные поля, специфичные для приложения

## Типы аккаунтов

### 1. User (Пользователи - клиенты)

**Назначение**: Обычные пользователи приложения, которые делают заказы, оставляют отзывы и бронируют столики.

**Хранится в Prisma:**

- `id` - внутренний UUID
- `keycloakId` - связь с Keycloak (уникальный)
- `email` - адрес электронной почты (уникальный, синхронизируется с Keycloak)
- `firstName` - имя
- `lastName` - фамилия
- `phone` - телефон (опционально)
- `avatar` - URL аватара (опционально)
- `balance` - баланс пользователя (Decimal, по умолчанию 0)
- `createdAt` - дата создания
- `updatedAt` - дата обновления

**Связи:**

- `orders` - заказы пользователя
- `reviews` - отзывы пользователя
- `bookings` (Appointment) - бронирования пользователя

**Хранится в Keycloak:**

- `keycloakId` - уникальный ID
- `email` - адрес электронной почты
- `password` - хешированный пароль

**Как создается:**

1. При регистрации через `POST /auth/register` создается запись в Keycloak
2. Затем создается запись в Prisma с полученным `keycloakId`
3. При логине через `POST /auth/login` запись в Prisma создается/обновляется автоматически

### 2. WorkerAccount (Аккаунты работников)

**Назначение**: Аккаунты сотрудников системы (администраторы, работники кофеен).

**Хранится в Prisma:**

- `id` - внутренний UUID
- `keycloakId` - связь с Keycloak (уникальный)
- `email` - адрес электронной почты (уникальный, синхронизируется с Keycloak)
- `firstName` - имя
- `lastName` - фамилия
- `role` - роль работника (enum WorkerRole):
  - `SYSTEM_ADMIN` - системный администратор
  - `BRAND_ADMIN` - администратор бренда
  - `CAFE_ADMIN` - администратор кофейни
  - `WORKER` - обычный работник
- `brandId` - ID бренда (опционально, для BRAND_ADMIN и связанных ролей)
- `cafeId` - ID кофейни (опционально, для CAFE_ADMIN и WORKER)
- `createdAt` - дата создания
- `updatedAt` - дата обновления

**Связи:**

- `brand` - бренд (если указан brandId)
- `cafe` - кофейня (если указан cafeId)

**Хранится в Keycloak:**

- `keycloakId` - уникальный ID
- `email` - адрес электронной почты
- `password` - хешированный пароль

**Как создается:**

1. При регистрации через `POST /auth/workers` создается запись в Keycloak
2. Затем создается запись в Prisma с полученным `keycloakId` и указанием роли
3. При логине через `POST /auth/login` запись в Prisma создается/обновляется автоматически
4. Проверяется, что email не занят как User (один email = один аккаунт)

## Различия между User и WorkerAccount

| Параметр                   | User                             | WorkerAccount                                 |
| -------------------------- | -------------------------------- | --------------------------------------------- |
| **Назначение**             | Клиенты приложения               | Сотрудники системы                            |
| **Роли**                   | Нет ролей                        | SYSTEM_ADMIN, BRAND_ADMIN, CAFE_ADMIN, WORKER |
| **Баланс**                 | Есть (balance)                   | Нет                                           |
| **Телефон**                | Есть (phone)                     | Нет                                           |
| **Аватар**                 | Есть (avatar)                    | Нет                                           |
| **Связи с брендом**        | Нет                              | Есть (brandId)                                |
| **Связи с кофейней**       | Нет (только через заказы/отзывы) | Есть (cafeId)                                 |
| **Связи с заказами**       | Есть (orders)                    | Нет                                           |
| **Связи с отзывами**       | Есть (reviews)                   | Нет                                           |
| **Связи с бронированиями** | Есть (bookings)                  | Нет                                           |

## Синхронизация данных

### При регистрации/логине

- Используется подход **dual-write** (запись в оба хранилища одновременно)
- Сначала создается/проверяется пользователь в Keycloak
- Затем создается/обновляется запись в Prisma с полученным `keycloakId`

### При обновлении email в Keycloak

- Webhook `POST /auth/webhook/keycloak` синхронизирует email в Prisma
- Обновляются только поля, которые хранятся в Keycloak (email)

### При обновлении бизнес-данных

- Изменения в Prisma (firstName, lastName, phone, avatar, balance и т.д.) не синхронизируются с Keycloak
- Keycloak хранит только данные для аутентификации

## Текущая реализация

### Реализовано

- ✅ Создание и управление **User** через `POST /auth/register` и `POST /auth/login`
- ✅ Создание и управление **WorkerAccount** через `POST /auth/workers` и `POST /auth/login`
- ✅ Разделение эндпоинтов: User и WorkerAccount имеют отдельные эндпоинты регистрации
- ✅ Проверка уникальности email: один email не может быть одновременно User и WorkerAccount
- ✅ Получение и обновление профиля работника (`GET /auth/workers/me`, `PATCH /auth/workers/me`)
- ✅ Синхронизация email между Keycloak и Prisma
- ✅ Автоматическое создание записи в Prisma при первом логине
- ✅ Webhook для синхронизации данных из Keycloak
- ✅ Правильная архитектура без циклических зависимостей
- ✅ WorkersModule с полной функциональностью

## Роли в системе

Роли определяются через enum `WorkerRole` в Prisma и хранятся только в базе данных:

- **SYSTEM_ADMIN** - полный доступ ко всей системе, управление брендами
- **BRAND_ADMIN** - управление брендом и всеми кофейнями бренда
- **CAFE_ADMIN** - управление конкретной кофейней
- **WORKER** - базовые права работника кофейни

**Важно**: В данный момент роли в Keycloak не настроены. Роли хранятся только в Prisma в поле `WorkerAccount.role`. Для полноценной авторизации потребуется синхронизация ролей между Prisma и Keycloak Realm Roles.
