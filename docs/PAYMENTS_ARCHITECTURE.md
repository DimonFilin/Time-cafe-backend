# Архитектура платежей, балансов и карт

## Принцип разделения ответственности

**Keycloak НЕ предназначен для хранения финансовых данных.**

Keycloak - это система аутентификации и авторизации. Она должна хранить только:

- Учетные данные (email, password)
- Роли и права доступа
- Сессии и токены

**Все финансовые данные должны храниться в Prisma/PostgreSQL:**

- Баланс пользователя
- Банковские карты
- История транзакций
- Платежные операции

### 2. Банковские карты

**Модель PaymentCard:**

```prisma
model PaymentCard {
  id            String   @id @default(uuid())
  userId        String
  user          User     @relation(fields: [userId], references: [id])

  // Безопасное хранение данных карты
  last4Digits   String   // Последние 4 цифры (например, "1234")
  cardType      String   // "visa", "mastercard", "mir"
  expiryMonth   Int      // Месяц истечения (1-12)
  expiryYear    Int      // Год истечения

  // Токен от платежного провайдера (Stripe, YooKassa и т.д.)
  providerToken String?  // Токен для повторных платежей (не храним полные данные карты!)

  // Метаданные
  isDefault     Boolean  @default(false) // Карта по умолчанию
  isActive      Boolean  @default(true)
  holderName    String?  // Имя держателя карты

  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
  deletedAt     DateTime? // Soft delete для карт

  transactions  Transaction[]

  @@map("payment_cards")
}
```

**Важно:**

- НЕ храним полный номер карты, CVV, PIN
- Используем токенизацию через платежные провайдеры (Stripe, YooKassa, CloudPayments)
- Храним только последние 4 цифры для отображения пользователю
- Полные данные карты обрабатываются только платежным провайдером

**Эндпоинты:**

- `GET /users/cards` - список карт пользователя
- `POST /users/cards` - добавление новой карты (через платежный провайдер)
- `DELETE /users/cards/:id` - удаление карты
- `PATCH /users/cards/:id/set-default` - установка карты по умолчанию

### 3. История транзакций

**Модель Transaction:**

```prisma
enum TransactionType {
  TOP_UP        // Пополнение баланса
  WITHDRAWAL    // Списание (заказ, возврат)
  REFUND        // Возврат средств
  PAYMENT       // Платеж картой
}

enum TransactionStatus {
  PENDING       // Ожидает обработки
  PROCESSING    // В процессе
  COMPLETED     // Успешно завершена
  FAILED        // Ошибка
  CANCELLED     // Отменена
}

model Transaction {
  id              String            @id @default(uuid())
  userId          String
  user            User              @relation(fields: [userId], references: [id])

  // Тип и статус
  type            TransactionType
  status          TransactionStatus @default(PENDING)

  // Сумма
  amount          Decimal           @db.Decimal(10, 2) // Положительная для TOP_UP, отрицательная для WITHDRAWAL
  currency        String            @default("RUB")

  // Связи
  orderId         String?           // Если транзакция связана с заказом
  order           Order?            @relation(fields: [orderId], references: [id])
  cardId          String?            // Если использовалась карта
  card            PaymentCard?      @relation(fields: [cardId], references: [id])

  // Платежный провайдер
  provider        String?            // "stripe", "yookassa", "cloudpayments"
  providerTransactionId String?     // ID транзакции в системе провайдера

  // Метаданные
  description     String?            // Описание транзакции
  metadata        Json?              // Дополнительные данные (JSON)

  // Баланс до и после
  balanceBefore   Decimal            @db.Decimal(10, 2)
  balanceAfter    Decimal            @db.Decimal(10, 2)

  createdAt       DateTime           @default(now())
  updatedAt       DateTime           @updatedAt

  @@index([userId, createdAt])
  @@index([status])
  @@map("transactions")
}
```

**Эндпоинты:**

- `GET /users/transactions` - история транзакций (с пагинацией)
- `GET /users/transactions/:id` - детали конкретной транзакции
- `POST /users/transactions/top-up` - пополнение баланса (создает транзакцию)

## Интеграция с платежными провайдерами

### Рекомендуемые провайдеры для России:

1. **YooKassa** (бывший Яндекс.Касса) - популярный в РФ
2. **CloudPayments** - российский провайдер
3. **Stripe** - международный (если нужна поддержка зарубежных карт)

### Процесс добавления карты:

1. Пользователь вводит данные карты в мобильном приложении
2. Данные отправляются напрямую в платежный провайдер (НЕ через наш бэкенд!)
3. Провайдер возвращает токен карты
4. Бэкенд сохраняет токен и метаданные (последние 4 цифры, тип карты)
5. Полные данные карты никогда не попадают на наш сервер

### Процесс пополнения баланса:

1. Пользователь выбирает карту и сумму
2. Бэкенд создает транзакцию со статусом PENDING
3. Отправляет запрос в платежный провайдер с токеном карты
4. Провайдер обрабатывает платеж
5. Webhook от провайдера обновляет статус транзакции
6. При успехе - обновляется баланс пользователя

## Безопасность

### Обязательные меры:

1. **Шифрование данных в покое** - PostgreSQL с включенным шифрованием
2. **HTTPS для всех запросов** - обязательное шифрование трафика
3. **PCI DSS compliance** - если обрабатываем карты напрямую (лучше использовать токенизацию)
4. **Аудит транзакций** - логирование всех операций с балансом
5. **Rate limiting** - ограничение частоты запросов на пополнение
6. **Валидация сумм** - проверка минимальных/максимальных сумм
7. **Двухфакторная аутентификация** - для операций с балансом (опционально)

## Почему НЕ Keycloak для финансовых данных?

1. **Keycloak не предназначен для этого** - это система аутентификации, не платежная система
2. **Нет поддержки транзакций** - Keycloak не имеет механизмов для финансовых операций
3. **Нет аудита** - нет встроенной системы логирования финансовых операций
4. **Производительность** - Keycloak оптимизирован для аутентификации, не для финансовых запросов
5. **Соответствие стандартам** - Keycloak не соответствует PCI DSS для хранения данных карт

### Логика работы

**Добавление карты:**

1. Пользователь отправляет данные карты (номер, срок действия, CVV)
2. Валидация данных (формат, срок действия)
3. Определение типа карты по номеру
4. Генерация симулированного токена провайдера
5. Сохранение только последних 4 цифр и метаданных
6. Первая карта автоматически становится картой по умолчанию

**Создание платежа:**

1. Пользователь выбирает карту и указывает сумму
2. Создается транзакция со статусом PENDING
3. Симуляция обработки платежа (500ms задержка)
4. Транзакция переходит в PROCESSING, затем COMPLETED
5. Генерируется providerTransactionId (симуляция Stripe)

**Возврат средств:**

1. Пользователь запрашивает возврат для завершенного платежа
2. Проверка, что оригинальная транзакция завершена
3. Проверка суммы возврата (не больше оригинала)
4. Создание транзакции типа REFUND
5. Автоматическая обработка возврата

### Важные особенности

1. **Нет хранения баланса** - оплата идет напрямую с карты
2. **Симуляция Stripe** - валидация = успех, без реальных запросов
3. **Токенизация** - полные данные карты не хранятся
4. **Soft delete** - карты помечаются как удаленные, не удаляются физически
5. **Аудит** - все транзакции сохраняются в истории
